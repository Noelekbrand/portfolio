Overview

The goal was to build a realistic HIL setup — the kind of workflow used in automotive and industrial automation — on accessible hardware. An Arduino Leonardo runs embedded firmware with a proper state machine, communicates over UART, and controls three LEDs via PWM with smooth brightness ramping.

A Raspberry Pi 3 acts as the external test master. It sends commands, reads ACK/NACK responses, polls for state transitions, and runs a full automated test suite in Python. No manual intervention needed.

Arduino Leonardo

  • ATmega32U4 microcontroller
  • 3 PWM channels — pins 9, 10, 11
  • 2 red LEDs + 1 green LED, 330Ω resistors
  • Soft brightness ramping in firmware
  • millis() based non-blocking timer
  • UART at 9600 baud

Raspberry Pi 3

  • Host and HIL test master
  • controller.py — communication library
  • test_controller.py — 61 test cases
  • CLI: python3 controller.py SET_RED1 75
  • Connected via USB serial

Firmware state machine

The firmware is structured as a deterministic four-state machine. Each state has clearly defined entry conditions, exit conditions, and allowed commands — making behavior predictable and testable.

IDLE
→
RAMPING
→
STEADY
⇄
ERROR
  • IDLE Waiting for commands. All PWM outputs off.
  • RAMPING Actively transitioning to a new brightness target.
  • STEADY Target brightness reached. Holding or running sequence.
  • ERROR Invalid command or out-of-range value received. Responds with NACK.

UART command protocol

Commands are sent as plain-text strings over serial. The firmware parses each line, validates the argument range, updates internal state, and responds with ACK or NACK.

Available commands
# Set individual channel brightness (0–255)
SET_RED1 <value>      → ACK or NACK
SET_RED2 <value>      → ACK or NACK
SET_GREEN <value>     → ACK or NACK

# Global brightness multiplier
SET_BRIGHTNESS <value>

# Autonomous traffic light sequence
SEQUENCE START
SEQUENCE STOP

# Query current state
STATUS               → STATE:IDLE | RAMPING | STEADY | ERROR
Python — controller.py example
# Run a command from the CLI
$ python3 controller.py SET_RED1 75
ACK

# Or use the library in test code
from controller import open_port, send_command, poll_until_steady

port = open_port("/dev/ttyACM0")
send_command(port, "SET_GREEN", 200)
poll_until_steady(port)

HIL testing and validation

The test suite covers command parsing, state transitions, PWM behavior, error handling, and sequence mode. Each test sends a command over serial and asserts the response and subsequent state — fully automated, no manual intervention.

What is tested

  • SET_RED1 / SET_RED2 / SET_GREEN with valid and out-of-range values
  • ACK on valid commands, NACK on invalid arguments
  • State transitions: IDLE → RAMPING → STEADY
  • Sequence start/stop and autonomous timing
  • STATUS response format and contents
  • SET_BRIGHTNESS global multiplier effect

Engineering insights

Designing the UART protocol first — before writing any test code — made the system much more testable. Having explicit ACK/NACK responses meant the test suite could be purely black-box: no internal state inspection needed.

Using millis() instead of delay() in firmware was essential. It kept the UART parser responsive during ramp transitions, which is a common pitfall in embedded projects.

This project mirrors real-world HIL workflows used in automotive ECU development, where an embedded controller is validated by an external test system before any deployment to hardware.
